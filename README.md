با یک دستور، بین سرور ایران و خارج یک **تانل لایه ۳ پایدار و خودترمیم** بسازید؛ پورت فوروارد کنید، با پنل‌هایی مثل **3x-ui / Hiddify / Marzban** وصلش کنید و با موتور بهینه‌سازی **High-Load** خیال‌تان از شبکه‌های پرترافیک راحت باشد — همه در **یک فایل**، بدون حتی یک سرویس مزاحم در پس‌زمینه.

## 🎯 چرا GRE Tunnel Manager؟

| | مزیت | توضیح |
| --- | --- | --- |
| 🪶 | **فوق سبک** | هیچ daemon و تایمر اضافه‌ای نصب نمی‌کند؛ همه‌چیز در کرنل لینوکس انجام می‌شود — مصرف CPU و RAM اسکریپت عملاً **صفر** است |
| 👥 | **مقیاس سازمانی** | پروفایل اختصاصی High-Load برای **۱۰ هزار تا +۱۰۰ هزار کاربر** (منوی ۱۴) |
| 🔧 | **خودترمیم** | Watchdog هوشمند: ۳ پینگ ناموفق پشت سر هم → ریستارت خودکار تانل |
| ⚡ | **سرعت بالا** | BBR + FQ + MSS Clamping خودکار + پخش بار روی همه هسته‌های CPU (RPS) |
| 🛡 | **ضدقطعی** | همه تنظیمات با systemd ذخیره می‌شوند و بعد از **ریبوت** خودکار برمی‌گردند |
| 🎛 | **همه‌کاره** | تانل استاندارد + Simple GRE + پورت‌فورواردینگ پیشرفته + رله Realm (Rust) |
| 🔥 | **فایروال هوشمند** | باز کردن خودکار GRE و FORWARD در iptables/ip6tables/UFW — حتی کنار Docker |
| 🌐 | **دوال استک** | پشتیبانی کامل IPv4 و IPv6 (ip6gre) با تشخیص خودکار |

## 📋 پیش‌نیازها

- سرور **Ubuntu 18.04+ / Debian 10+** (با systemd) — هم ایران، هم خارج
- دسترسی **root**
- باز بودن **پروتکل GRE (پروتکل شماره 47)** در فایروال دیتاسنتر (Security Group)

## 📦 نصب و اجرا

دستور زیر را در **هر دو سرور** (ایران و خارج) اجرا کنید:

```bash
bash <(curl -fsSL --ipv4 https://raw.githubusercontent.com/Sir-Adnan/GRE-Tunnel-Manager/main/gre.sh)
```

💿 در اولین اجرا، اسکریپت پیشنهاد می‌کند میانبر `igre` را نصب کند — از آن به بعد فقط تایپ کنید:

```bash
igre
```

bash <(curl -fsSL --ipv4 https://raw.githubusercontent.com/Sir-Adnan/GRE-Tunnel-Manager/main/gre-stable.sh)

</details>

## 🚀 ساخت تانل در ۳ قدم

### قدم ۱ — سرور خارج 🌍
گزینه **[1] Kharej Server** را بزنید → آی‌پی **سرور ایران** را وارد کنید → یک **Tunnel ID** انتخاب کنید (مثلاً `10`) → روی سؤال MTU فقط **Enter** بزنید.

### قدم ۲ — سرور ایران 🇮🇷
گزینه **[2] Iran Server** را بزنید → آی‌پی **سرور خارج** را وارد کنید → **همان Tunnel ID** (`10`) → **Enter** برای MTU.

### قدم ۳ — تمام! ✅
اسکریپت آی‌پی‌های داخلی را نشان می‌دهد:

| سرور | آی‌پی داخلی تانل |
| --- | --- |
| خارج | `10.0.10.1` |
| ایران | `10.0.10.2` |

تست اتصال از سرور ایران:

```bash
ping 10.0.10.1
```

💡 حالا در پنل خود (3x-ui / Hiddify / Marzban) آی‌پی داخلی سمت مقابل را به عنوان مقصد بدهید، یا با **گزینه ۹** پورت فوروارد کنید.

> ⚠️ دو قانون طلایی: **Tunnel ID یکسان** + **نسخه اسکریپت یکسان** در هر دو سرور.

## 🔀 آموزش پورت فورواردینگ (گزینه ۹)

بعد از ساخت تانل، روی **سرور ایران** گزینه **[9]** را بزنید:

| سؤال | چه چیزی وارد کنیم | مثال |
| --- | --- | --- |
| **Local Port** | پورتی که کاربران به آن وصل می‌شوند | `443` |
| **Dest Tunnel IP** | آی‌پی داخلی تانلِ **سرور خارج** | `10.0.10.1` |
| **Dest Port** | پورت سرویس شما روی سرور خارج | `443` |

از این لحظه، هر اتصالی به `پورت 443 سرور ایران` مستقیم از داخل تانل به سرور خارج می‌رسد. 🚀

> 🛡 **محافظ SSH:** اگر پورتی را وارد کنید که سرویس محلی (مثل SSH) رویش فعال است، اسکریپت هشدار می‌دهد تا ناخواسته دسترسی خودتان را قطع نکنید.

> ♻️ قوانین بعد از ریبوت خودکار برمی‌گردند. حذف: گزینه **11** — ویرایش: گزینه **10**.

## ⚡ حالت High-Load — شبکه‌های ۱۰ هزار تا +۱۰۰ هزار کاربر (منوی ۱۴)

اگر ترافیک سنگین دارید، این منو سه گلوگاه اصلی لینوکس را با **تنظیمات صرفاً کرنلی** (بدون هیچ پروسه جدید!) برطرف می‌کند:

| زیرگزینه | کاربرد |
| --- | --- |
| **[1] Performance Status** | نمایش زنده: درصد پر شدن جدول اتصالات (conntrack)، وضعیت RPS، ترافیک RX/TX هر تانل |
| **[2] Apply High-Load Profile** | فعال‌سازی پروفایل در دو سطح: `HIGH` (۱۰–۵۰ هزار) / `EXTREME` (+۱۰۰ هزار) |
| **[3] Remove Profile** | برگشت کامل به تنظیمات پیش‌فرض |
| **[4] Scaling Guide** | راهنمای مقیاس‌پذیری داخل خود اسکریپت |

**پروفایل چه می‌کند؟**
- 🧠 **RPS**: ترافیک تانل به‌طور پیش‌فرض فقط روی **یک هسته CPU** می‌افتد؛ پروفایل آن را روی همه هسته‌ها پخش می‌کند
- 📊 **بزرگ‌سازی جدول conntrack**: تا ۱,۰۴۸,۵۷۶ اتصال همزمان + حذف سریع‌تر اتصالات مرده
- 🚄 **NOTRACK**: پکت‌های حامل GRE از مسیر conntrack خارج می‌شوند → CPU کمتر به ازای هر پکت
- 🔌 بازه کامل پورت، بافرهای بزرگ‌تر و TCP بهینه برای مسیرهای پر-RTT

**مصرف منابع (شفاف و صادقانه):**

| حالت | CPU | RAM |
| --- | --- | --- |
| اسکریپت + منوها | ~۰٪ | ~۰ |
| هر تانل + Watchdog | ~۰٪ | ~۲ MB |
| پروفایل HIGH | تغییری نمی‌کند (per-core حتی کمتر) | فقط با اتصال واقعی رشد می‌کند — سقف ~۱۸۰ MB |
| پروفایل EXTREME | تغییری نمی‌کند | سقف ~۴۰۰ MB (پیشنهاد: سرور ۲+ گیگ) |

> 💡 **نکته طلایی برای +۵۰ هزار کاربر:** لینوکس به ازای هر «مقصد» حدود ۶۴ هزار اتصال همزمان NAT اجازه می‌دهد. کافیست با گزینه ۹ **دو تا چهار پورت** به همان سرویس خارج فوروارد کنید (مثلاً 443 و 8443 و 2053) و کاربران را در پنل بین آن‌ها تقسیم کنید — هر پورت اضافه یعنی ۶۴ هزار ظرفیت جدید. 📈

## 🛠 نکات حرفه‌ای

- **MTU سفارشی:** موقع ساخت تانل می‌توانید به‌جای پیش‌فرض امن `1430`، روی مسیرهای تمیز تا `1472` (IPv4) یا `1452` (IPv6) بروید تا چند درصد سرعت بیشتر بگیرید. مطمئن نیستید؟ فقط Enter بزنید.
- **چند تانل همزمان:** تانل‌ها `key`دار هستند؛ می‌توانید بین همان دو سرور چند تانل با **Tunnel ID های متفاوت** بسازید.
- **IPv6:** اگر آی‌پی ریموت را IPv6 وارد کنید، اسکریپت خودکار تانل ip6gre می‌سازد.
- **Realm (منوی ۱۳):** وقتی به‌جای تانل لایه ۳ فقط یک رله TCP/UDP سبک می‌خواهید — با مصرف چند مگابایت RAM.
- **توقف موقت تانل:** `systemctl stop gre-tun-10` — نگران نباشید، Watchdog دیگر آن را بی‌اجازه روشن نمی‌کند؛ برای شروع دوباره: `systemctl start gre-tun-10`

## ❓ سوالات متداول

<details>
<summary><b>تانل وصل نمی‌شود؟</b></summary>

1. **Tunnel ID** در هر دو سرور یکسان است؟
2. **نسخه اسکریپت** در هر دو سرور یکسان است؟ (نسخه جدید با Stable قدیمی ناسازگار است)
3. فایروال دیتاسنتر **پروتکل GRE (47)** را باز گذاشته؟ (این در پنل دیتاسنتر است، نه داخل سرور)
4. وضعیت سرویس: `systemctl status gre-tun-10`
</details>

<details>
<summary><b>مصرف CPU و RAM چقدر است؟</b></summary>

خود اسکریپت فقط موقع باز بودن منو زنده است (~۴ MB). هر تانل یک Watchdog دارد (~۲ MB و ~۰٪ CPU). خود GRE در کرنل اجرا می‌شود و رمزنگاری ندارد — یعنی سبک‌ترین نوع تانل ممکن. جزئیات دقیق: منوی ۱۴ → Performance Status.
</details>

<details>
<summary><b>سرعت کم است یا بعضی سایت‌ها باز نمی‌شوند؟</b></summary>

- از v13.8 به بعد **MSS Clamping خودکار** فعال است که رایج‌ترین علت این مشکل (خرابی PMTUD) را حل می‌کند — مطمئن شوید نسخه‌تان به‌روز است.
- افت ذاتی GRE فقط ~۳ تا ۷٪ است؛ اگر افت شدید می‌بینید معمولاً مربوط به کیفیت مسیر بین‌الملل یا محدودسازی پروتکل GRE توسط اپراتور است، نه اسکریپت.
- برای ترافیک بالا حتماً پروفایل High-Load (منوی ۱۴) را فعال کنید.
</details>

<details>
<summary><b>بعد از ریبوت سرور چه می‌شود؟</b></summary>

هیچ! همه‌چیز (تانل، فورواردینگ، فایروال، پروفایل Performance) با systemd ذخیره شده و خودکار برمی‌گردد.
</details>

<details>
<summary><b>سرویس Watchdog چیست؟</b></summary>

هر ۱۰ ثانیه سمت مقابل تانل را پینگ می‌کند؛ اگر ۳ پینگ پشت سر هم جواب نگیرد، تانل را ریستارت می‌کند. اگر خودتان تانل را stop کنید، Watchdog هم هوشمندانه متوقف می‌شود.
</details>

<details>
<summary><b>چطور اسکریپت را آپدیت کنم؟</b></summary>

گزینه **[5] Re-install Shortcut** آخرین نسخه را دانلود و جایگزین می‌کند. بعد از آپدیت، اگر تانل قدیمی دارید بهتر است یک‌بار در هر دو سرور تانل را از نو بسازید (گزینه ۱/۲ با همان ID — خودش overwrite می‌کند).
</details>

## 🧯 عیب‌یابی سریع

```bash
systemctl status gre-tun-10        # وضعیت سرویس تانل (به جای 10، آی‌دی خودتان)
ping 10.0.10.1                     # تست اتصال داخلی از ایران به خارج
ip -d link show gre-out-10         # مشخصات اینترفیس (سمت ایران)
journalctl -u gre-tun-10 -n 20     # لاگ‌های سرویس
iptables -t nat -L PREROUTING -n   # لیست قوانین فورواردینگ
cat /proc/sys/net/netfilter/nf_conntrack_count   # تعداد اتصالات فعلی NAT
```
